Plan-Driven vs. Agile

Two Philosophies of Development

Most software processes can be categorized into two main approaches: plan-driven and agile. In practice, many projects use a hybrid approach that incorporates elements from both.

Plan-driven Processes

  • All activities are planned in advance.
  • Progress is measured against the plan.
  • Works well for large systems where requirements are well-understood and stable.
  • Helps coordinate work across multiple development sites.

Agile Processes

  • Planning is incremental and flexible.
  • Easier to respond to changing customer requirements.
  • Promotes rapid delivery of useful software.
  • Relies on close customer collaboration.

The Waterfall Model

The Classic Sequential Approach

The Waterfall model is a plan-driven process where development flows sequentially from one phase to the next, like a waterfall. A phase cannot begin until the previous one is complete.

Limitations

The main drawback of the Waterfall model is its inflexibility. It is very difficult and costly to respond to changing customer requirements once a phase is complete, making it unsuitable for projects where requirements are likely to change.

Incremental Development

Building Software Piece by Piece

Incremental development combines elements of all development activities. The software is specified, designed, built, and tested in a series of increments, with each increment adding more functionality.

  • Customer feedback: Easier to get feedback on work that has been done.
  • Rapid delivery: Customers get useful software earlier than with a waterfall process.

  • System structure degrades: The code can become messy as new increments are added without refactoring.
  • Contracts: May be difficult if the contract requires a complete system specification upfront.

Reuse-Oriented Engineering

Don't Reinvent the Wheel

In most software projects, some reuse is common. This approach is based on assembling systems from existing, reusable components rather than developing everything from scratch. This can significantly reduce development time and cost.

Types of Reusable Components:

  • Web services that can be called remotely.
  • Object collections developed for frameworks like .NET or J2EE.
  • COTS (Commercial Off-The-Shelf) systems that can be configured for a specific environment.
